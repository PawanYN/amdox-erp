2b. HOW PROJECT MANAGEMENT CONNECTS TO HR · SCM · FINANCE

PM doesn't do the work itself — it coordinates the other three around a project. HR supplies people, SCM supplies materials, Finance supplies/tracks money.

🟢 PM ← HR  (people)

PM assigns team members to project tasks → reads from HR's Employee table. "Resource allocation" = picking which employees work on which task.

The utilisation heatmap shows if someone is overloaded (on too many projects at once).

HR → PM project.resource_needed
🟠 PM → SCM  (materials)

A project often needs physical items (e.g. construction needs cement, steel). PM signals the need → SCM raises the PO and procures it.

This is exactly when a manual requisition gets raised — the project is the reason.

PM → SCM project.material_requested
⇄
PM Project Management (the coordinator)
⇄
🟦 PM ↔ Finance  (money)

Every project has a budget (planned spend).

As it runs, real costs pile up: salaries of assigned employees (HR → Finance payroll) + materials bought (SCM → Finance invoices).

Finance feeds those actual costs back → PM compares planned vs actual → fires variance alerts if over budget.

PM ⇄ Finance budget.set / cost.reported
One-line summary:   PM ← HR = staff the project  ·  PM → SCM = procure project supplies  ·  PM ⇄ Finance = track if the project stays profitable. The salary cost flows HR → Finance → PM, and material cost flows SCM → Finance → PM, so Finance is the meeting point where PM sees total real spend vs budget.
⚠ Architecture note (DDD): PM never queries HR / SCM / Finance databases directly. It talks to them only through domain events (e.g. PM emits project.resource_needed, HR responds). This keeps the modules loosely coupled — PM doesn't break if HR's internal schema changes.

5. SUMMARY
🔑 Login: Keycloak verifies identity → backend resolves tenant + role from UserRole.
📋 Plan: Low stock auto-raises PR → approved PO → vendor portal.
📦 Receive: Goods receipt updates inventory → triggers AP 3-way match.
💰 Post: Approved invoice → balanced GL journal entry (debit = credit).
🛒 Sell: Sales order → AR invoice → payment → bank reconcile.
👥 Parallel: HR payroll (saga) + PM (V2) both feed Finance.
⭐ Always-on: Forecasting, BI, Notification, Audit, Gateway.

24 tables, every one carries tenantId — row-level security at the Prisma middleware layer, no cross-tenant leakage.

Modules never query each other's DB — they talk only through domain events (EventEmitter2 + Outbox pattern).

😊 UUID keys (no IDOR) · soft-delete (GDPR) · hash-chained audit · saga rollback.
